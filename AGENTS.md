# AGENTS.md

## 1. 项目名称

机场区域大规模小型无人车蚁群仿真系统  
英文名称：Airport Ant Swarm Simulation

## 2. 项目目标

本项目研究强对抗、弱通信和节点失效条件下，大规模小型无人车群体的仿生协同机制。

最终目标是在二维机场区域中仿真最多1000个小型无人车节点，使节点仅依靠局部观测、邻域交互和语义信息素，实现：

- 机场周界持续巡逻；
- 跑道和滑行道巡检；
- 机库、塔台、油料区等关键点值守；
- 跑道异物搜索和局部封控；
- 异常目标发现、确认与局部增援；
- 节点失效后的自主补位；
- 通信受限条件下的任务维持；
- 道路封闭或风险变化后的路径重构。

项目重点不是制作高保真的机场动画，而是验证蚁群生物机制能否提升群体的：

- 自组织能力；
- 局部信息共享能力；
- 动态任务分工能力；
- 受扰恢复能力；
- 大规模扩展能力。

开始修改代码前，请先阅读 `PROJECT_BACKGROUND.md`。

## 3. 当前技术路线

项目按照以下机制链展开：

1. 蚁群生物机制建模；
2. 无人车行为基元表达；
3. 多通道语义信息素交互；
4. 响应阈值驱动的自主分工；
5. 局部招募与法定数确认；
6. 正负反馈耦合调控；
7. 受扰条件下的弹性补位与恢复；
8. 1000节点规模实验。

核心蚁群机制包括：

- 信息素沉积、扩散、挥发和强化；
- 个体任务响应阈值；
- 局部招募；
- 法定数触发；
- 密度与风险负反馈；
- 任务饱和抑制；
- 节点失效后的刺激累积和替代补偿。

## 4. 语义信息素通道

系统计划包含以下信息素通道：

- `task`：任务位置、紧迫度和需求强度；
- `path`：可通行方向和路径收益；
- `risk`：障碍、威胁、干扰和封锁；
- `density`：局部节点密度和拥塞程度；
- `coverage`：区域历史访问和巡逻空洞；
- `service`：值守、补位或中继服务缺口；
- `trust`：后期用于描述信息可信度。

信息素不只是绘图效果，而应实际进入节点决策过程。

## 5. 机场场景

第一版世界尺寸：

- 世界宽度：3000 m；
- 世界高度：3000 m；
- 二维连续空间；
- Pygame显示窗口：默认按主显示器桌面分辨率启动，允许用户缩放；窗口不得小于640 × 480像素。
- 地图应根据实际窗口尺寸等比例缩放并居中显示，不得因宽高比变化而拉伸变形。

机场主要区域包括：

- 主跑道；
- 平行滑行道和连接滑行道；
- 停机坪；
- 机库群；
- 塔台；
- 油料保障区；
- 机场出入口；
- 机场周界；
- 无人车集结与待命区。

机场地图是抽象实验地图，不复刻具体真实机场。

## 6. 个体模型原则

每个无人车节点至少应具有：

- 唯一编号；
- 二维位置；
- 航向；
- 线速度；
- 当前角色；
- 当前任务；
- 能量状态；
- 健康状态；
- 感知范围；
- 通信范围；
- 邻居集合；
- 各任务响应阈值；
- 本地信息素观测；
- 有限长度的局部历史信息。

节点策略不得直接访问以下全局信息：

- 全部节点的真实位置；
- 全局最优路径；
- 所有任务的完整状态；
- 全局信息素矩阵；
- 由中心节点计算好的任务分配结果。

全局状态只能由仿真器用于统计、可视化和评价，不能直接提供给节点决策器。

## 7. 初期无人车模型

初期采用轻量级二维运动学模型，不引入复杂轮胎动力学。

建议状态更新形式：

- 位置由速度和航向积分更新；
- 速度、转向角和角速度存在上限；
- 节点不能穿过不可通行区域；
- 碰撞处理与避障算法分离；
- 时间步长由配置文件统一控制。

高保真车辆模型和ROS接口属于后期扩展内容。

## 8. 开发阶段

当前采用逐步递增方式：

### 阶段1：机场基础地图

显示机场区域并支持世界坐标与屏幕坐标转换。

### 阶段2：无人车个体与初始部署

加入单车运动模型，并从20、50、200个节点逐步测试。

### 阶段3：基础巡逻

加入覆盖信息素和密度信息素。

### 阶段4：关键点值守

加入服务缺口、响应阈值和任务角色切换。

### 阶段5：突发事件

加入跑道异物、道路封闭、异常目标和节点失效。

### 阶段6：局部招募与法定数

实现多节点确认和群体任务启动。

### 阶段7：通信与感知扰动

加入丢包、时延、通信黑区和观测噪声。

### 阶段8：规模扩展

节点规模按以下顺序增加：

50 → 100 → 300 → 500 → 1000

不得在基础机制未经验证时直接编写1000节点复杂综合场景。

## 11. 开发环境

默认开发环境：

- Windows；
- Anaconda；
- Python 3.10；
- Pygame；
- NumPy；
- 后续可能加入 PyYAML、Matplotlib、Pandas 和 Pytest。

运行环境名称：

```text
ant_airport
```

基础运行命令：

```powershell
conda activate ant_airport
python main.py
```

安装基础依赖：

```powershell
pip install pygame numpy
```

新增第三方生产依赖前必须说明必要性，不要自行引入大型框架。

## 12. 编码规范

- 使用Python 3.10兼容语法；
- 所有公开类和函数使用类型注解；
- 核心数据结构优先使用`dataclass`；
- 文件和变量使用英文命名；
- 注释和解释优先使用中文，便于初学者阅读；
- 函数保持单一职责；
- 不在一个文件中堆积所有功能；
- 避免不必要的继承层次和设计模式；
- 避免过早优化；
- 随机过程必须支持固定随机种子；
- 参数不得大量硬编码在算法函数内部；
- 重要参数应集中到配置模块或配置文件；
- 对可能失败的文件操作和参数输入提供清晰错误信息；
- 不使用全局可变状态在模块之间隐式传递数据。

## 13. 性能要求

最终需要支持1000节点，因此从早期开始注意：

- 避免每个节点遍历全部其他节点；
- 邻居搜索后期应使用空间网格、空间哈希或其他局部索引；
- 信息素场优先使用NumPy数组批量更新；
- 避免在主循环中频繁创建大量临时对象；
- 仿真逻辑更新与Pygame绘制解耦；
- 可视化帧率与仿真时间步应允许分离；
- 任何性能优化必须先有测量数据，不能凭感觉重写。

在20至200节点阶段可以使用简单实现，但应保留后续替换邻居搜索模块的接口。

## 16. Codex工作方式

执行复杂任务前：

1. 先检查现有文件；
2. 用简短文字说明对项目的理解；
3. 给出修改计划；
4. 指出将修改或新增哪些文件；
5. 等用户同意或任务明确后再实施。

修改代码时：

- 优先进行小步、可运行的修改；
- 不要一次重构整个项目；
- 不要删除已有功能；
- 不要修改与当前任务无关的代码；
- 不确定需求时先提出关键问题；
- 发现现有设计问题时先解释，再调整；
- 每完成一个阶段，都应运行代码或测试；
- 不要声称测试通过，除非实际执行了测试命令。

完成任务后必须说明：

- 修改了哪些文件；
- 实现了什么；
- 使用什么命令运行；
- 实际执行了哪些测试；

## 17. 面向初学者的要求

用户正在逐步学习并完成项目，因此：

- 给出代码时同时说明模块作用；
- 对关键类、函数、数据流和算法进行解释；
- 不要只提供最终代码而不解释；
- 报错时先定位根因，再给修复步骤；
- 每次尽量只引入少量新概念；
- 复杂算法先给简化版本，再逐步增强；
- 命令需要明确说明在哪个终端和目录执行。

## 18. 禁止事项

除非用户明确要求，不得：

- 将二维项目直接改成ROS或Gazebo工程；
- 引入深度学习或强化学习框架；
- 使用集中式全局任务分配替代仿生机制；
- 为了视觉效果牺牲算法可验证性；
- 把全局真实状态泄露给无人车局部策略；
- 一次性生成全部最终系统；
- 编造已经完成的测试结果；
- 随意修改世界尺寸和机场区域定义；
- 复制真实机场的敏感布局；
- 将本项目改造成具体攻击或武器控制系统。

## 19. 当前任务的完成标准

当前地图阶段只有在以下条件满足时才算完成：

- `python main.py`可以正常运行；
- Pygame窗口可打开和退出；
- 窗口默认适配当前屏幕，并在用户缩放后保持地图比例和界面可用；
- 机场各功能区域正确显示；
- 世界坐标和屏幕坐标可以相互转换；
- 鼠标位置能显示对应世界坐标；
- 可以保存地图截图；
- 没有明显异常或持续报错；
- 代码结构可以支持后续加入无人车模型。
